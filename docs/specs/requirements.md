# Requirements Document - DocMind RAG System

## Introduction

DocMind is a professional private document reasoning system based on RAG (Retrieval-Augmented Generation) architecture. The system allows authenticated users to upload documents, transform them into vector representations through embeddings, and perform natural language queries that combine semantic retrieval with contextualized response generation.

The architecture consists of two main microservices:
- **Go_Service**: User management, authentication, file upload, and access control
- **Python_Service**: Complete RAG pipeline (chunking, embeddings, semantic retrieval, generation)

The system uses gRPC communication to minimize latency between services and is designed to run entirely on local infrastructure using only free tools.

## Glossary

- **DocMind_System**: The complete document reasoning system
- **Go_Service**: Go microservice responsible for authentication, users, and file management
- **Python_Service**: Python microservice (FastAPI) responsible for the RAG pipeline
- **User**: Authenticated system user
- **Document**: File uploaded by a user (PDF, TXT, DOCX, MD)
- **Chunk**: Text fragment extracted from a document for processing
- **Embedding**: Vector representation of a text chunk
- **Vector_Store**: Qdrant database for embedding storage
- **Object_Storage**: MinIO system for original file storage
- **JWT_Token**: JSON Web Token authentication token
- **RAG_Pipeline**: Complete Retrieval-Augmented Generation process
- **Semantic_Search**: Vector similarity search in the Vector_Store
- **LLM**: Large Language Model (local Ollama)
- **Query**: Natural language query made by a user
- **Context**: Retrieved chunks relevant to a query
- **Response**: Response generated by the LLM based on retrieved context
- **PostgreSQL_DB**: Relational database for metadata
- **gRPC_Interface**: Communication interface between microservices
- **REST_API**: HTTP interface for external clients

## Requirements

### Requirement 1: User Management

**User Story:** As a system administrator, I want to manage users, so that only authorized people can access the system.

#### Acceptance Criteria

1. THE Go_Service SHALL store user information in PostgreSQL_DB
2. WHEN a new user registers, THE Go_Service SHALL validate that the email is unique
3. WHEN a new user registers, THE Go_Service SHALL hash the password before storing it
4. THE Go_Service SHALL provide REST endpoints to create, read, update, and delete users
5. WHEN user information is requested, THE Go_Service SHALL omit the password hash in the response

### Requirement 2: Authentication and Authorization

**User Story:** As a user, I want to authenticate securely, so that my documents are protected.

#### Acceptance Criteria

1. WHEN a user provides valid credentials, THE Go_Service SHALL generate a JWT_Token
2. THE JWT_Token SHALL contain the user_id and expire after 24 hours
3. WHEN a user provides invalid credentials, THE Go_Service SHALL return a 401 error
4. THE Go_Service SHALL validate JWT_Tokens on all protected routes
5. WHEN a JWT_Token is invalid or expired, THE Go_Service SHALL return a 401 error
6. THE Go_Service SHALL extract the user_id from the JWT_Token for authorized operations

### Requirement 3: Document Upload

**User Story:** As an authenticated user, I want to upload documents, so that I can query them later.

#### Acceptance Criteria

1. THE Go_Service SHALL accept files with PDF, TXT, DOCX, and MD extensions
2. WHEN a user uploads a file with an unsupported extension, THE Go_Service SHALL return a 400 error
3. WHEN a user uploads a valid file, THE Go_Service SHALL store it in Object_Storage
4. WHEN a file is successfully stored, THE Go_Service SHALL create a record in PostgreSQL_DB with metadata (filename, user_id, upload_date, file_size, storage_path)
5. THE Go_Service SHALL generate a unique document_id for each uploaded document
6. WHEN a file is successfully stored, THE Go_Service SHALL invoke Python_Service via gRPC for indexing
7. WHEN the gRPC invocation fails, THE Go_Service SHALL mark the document as "pending_indexing" in PostgreSQL_DB

### Requirement 4: Document Access Control

**User Story:** As a user, I want only me to access my documents, so that I maintain my privacy.

#### Acceptance Criteria

1. WHEN a user requests access to a document, THE Go_Service SHALL verify that the document_id belongs to the user_id from the JWT_Token
2. WHEN a user attempts to access another user's document, THE Go_Service SHALL return a 403 error
3. THE Go_Service SHALL provide an endpoint to list documents for the authenticated user
4. WHEN a user deletes a document, THE Go_Service SHALL delete the file from Object_Storage and the record from PostgreSQL_DB
5. WHEN a user deletes a document, THE Go_Service SHALL invoke Python_Service via gRPC to delete embeddings from Vector_Store

### Requirement 5: Document Text Extraction

**User Story:** As a system, I want to extract text from different file formats, so that I can process them uniformly.

#### Acceptance Criteria

1. WHEN Python_Service receives a PDF file, THE Python_Service SHALL extract text using a PDF parsing library
2. WHEN Python_Service receives a TXT or MD file, THE Python_Service SHALL read the content as UTF-8 plain text
3. WHEN Python_Service receives a DOCX file, THE Python_Service SHALL extract text using a DOCX parsing library
4. WHEN text extraction fails, THE Python_Service SHALL return a descriptive error via gRPC
5. THE Python_Service SHALL preserve paragraph structure during extraction

### Requirement 6: Document Chunking

**User Story:** As a system, I want to split documents into manageable fragments, so that embeddings are more accurate and retrieval more effective.

#### Acceptance Criteria

1. WHEN Python_Service processes a document, THE Python_Service SHALL split it into chunks using LangChain text splitters
2. THE Python_Service SHALL configure a maximum chunk size of 1000 characters
3. THE Python_Service SHALL configure an overlap of 200 characters between consecutive chunks
4. THE Python_Service SHALL preserve sentence boundaries when possible
5. WHEN a document generates chunks, THE Python_Service SHALL assign a unique chunk_id and maintain sequential order

### Requirement 7: Embedding Generation

**User Story:** As a system, I want to convert text chunks into vectors, so that I can perform semantic searches.

#### Acceptance Criteria

1. THE Python_Service SHALL generate embeddings using a LangChain-compatible embedding model
2. WHEN Python_Service generates embeddings, THE Python_Service SHALL use a model that produces vectors of consistent dimension
3. THE Python_Service SHALL process chunks in batches to optimize performance
4. WHEN embedding generation fails for a chunk, THE Python_Service SHALL log the error and continue with the next chunks
5. THE Python_Service SHALL normalize embedding vectors before storing them

### Requirement 8: Vector Storage

**User Story:** As a system, I want to store embeddings in a vector database, so that I can perform efficient similarity searches.

#### Acceptance Criteria

1. THE Python_Service SHALL store embeddings in Vector_Store (Qdrant)
2. WHEN Python_Service stores an embedding, THE Python_Service SHALL include metadata (document_id, chunk_id, chunk_text, user_id)
3. THE Python_Service SHALL create a separate collection per user in Vector_Store
4. WHEN document indexing is complete, THE Python_Service SHALL return success via gRPC
5. THE Python_Service SHALL handle automatic reconnections if Vector_Store is temporarily unavailable

### Requirement 9: Semantic Search

**User Story:** As a user, I want to search information in my documents using natural language, so that I can find relevant content without knowing exact keywords.

#### Acceptance Criteria

1. WHEN a user submits a query, THE Python_Service SHALL generate a query embedding using the same model as documents
2. THE Python_Service SHALL perform cosine similarity search in Vector_Store
3. THE Python_Service SHALL retrieve the top 5 most similar chunks by default
4. THE Python_Service SHALL filter results to include only chunks from the requesting user_id
5. WHEN no relevant chunks are found with similarity greater than 0.7, THE Python_Service SHALL return an empty list

### Requirement 10: RAG Response Generation

**User Story:** As a user, I want to receive contextualized answers to my questions, so that I can get accurate information from my documents.

#### Acceptance Criteria

1. WHEN Python_Service retrieves relevant chunks, THE Python_Service SHALL construct a prompt that includes the query and context
2. THE Python_Service SHALL send the prompt to the LLM (Ollama) for generation
3. THE Python_Service SHALL configure a 30-second timeout for LLM generation
4. WHEN the LLM generates a response, THE Python_Service SHALL return it along with the source chunks
5. WHEN the LLM is unavailable, THE Python_Service SHALL return a 503 error
6. THE Python_Service SHALL include references to the document_ids of the chunks used in the response

### Requirement 11: Go Service REST API

**User Story:** As a frontend developer, I want a well-documented REST API, so that I can integrate the system easily.

#### Acceptance Criteria

1. THE Go_Service SHALL expose REST endpoints for authentication (POST /auth/login, POST /auth/register)
2. THE Go_Service SHALL expose REST endpoints for user management (GET /users/me, PUT /users/me, DELETE /users/me)
3. THE Go_Service SHALL expose REST endpoints for documents (POST /documents, GET /documents, GET /documents/:id, DELETE /documents/:id)
4. THE Go_Service SHALL expose a REST endpoint for queries (POST /query)
5. THE Go_Service SHALL return responses in JSON format
6. THE Go_Service SHALL include appropriate HTTP status codes (200, 201, 400, 401, 403, 404, 500, 503)
7. WHEN an error occurs, THE Go_Service SHALL return a descriptive error message in JSON format

### Requirement 12: gRPC Interface Between Services

**User Story:** As a system architect, I want efficient communication between microservices, so that it minimizes latency in critical operations.

#### Acceptance Criteria

1. THE Go_Service SHALL define gRPC services for IndexDocument and QueryDocument in proto files
2. THE Python_Service SHALL implement the gRPC server with the defined services
3. THE Go_Service SHALL act as a gRPC client to invoke Python_Service
4. WHEN Go_Service invokes IndexDocument, THE gRPC_Interface SHALL transmit (document_id, user_id, file_path, file_type)
5. WHEN Go_Service invokes QueryDocument, THE gRPC_Interface SHALL transmit (query_text, user_id, top_k)
6. THE gRPC_Interface SHALL return structured responses with status codes and error messages
7. THE Go_Service SHALL configure a 60-second timeout for indexing gRPC calls
8. THE Go_Service SHALL configure a 45-second timeout for query gRPC calls

### Requirement 13: Configuration and Secrets Management

**User Story:** As a security engineer, I want credentials never uploaded to the repository, so that we maintain system security.

#### Acceptance Criteria

1. THE DocMind_System SHALL use .env files to store sensitive configuration
2. THE DocMind_System SHALL include .env in .gitignore
3. THE DocMind_System SHALL provide .env.example files with configuration variables without real values
4. THE Go_Service SHALL read configuration from environment variables (DATABASE_URL, JWT_SECRET, MINIO_ENDPOINT, GRPC_PYTHON_SERVICE_URL)
5. THE Python_Service SHALL read configuration from environment variables (QDRANT_URL, OLLAMA_URL, GRPC_PORT)
6. WHEN a required environment variable is not defined, THE DocMind_System SHALL fail to start with a clear error message

### Requirement 14: Dockerized Infrastructure

**User Story:** As a developer, I want to run the entire system with a single command, so that I can develop and test easily.

#### Acceptance Criteria

1. THE DocMind_System SHALL provide a docker-compose.yml file that defines all services
2. THE docker-compose.yml SHALL include services for Go_Service, Python_Service, PostgreSQL_DB, Vector_Store, Object_Storage, and LLM
3. THE docker-compose.yml SHALL configure networks for inter-service communication
4. THE docker-compose.yml SHALL configure volumes for data persistence
5. WHEN a developer runs "docker-compose up", THE DocMind_System SHALL start all services in the correct order
6. THE docker-compose.yml SHALL configure health checks for critical services
7. THE docker-compose.yml SHALL expose only necessary ports to the host (Go_Service REST API)

### Requirement 15: Error Handling and Logging

**User Story:** As a system operator, I want detailed logs, so that I can diagnose problems quickly.

#### Acceptance Criteria

1. THE Go_Service SHALL log all HTTP requests with timestamp, method, path, and response code
2. THE Python_Service SHALL log all gRPC invocations with timestamp and parameters
3. WHEN an error occurs, THE DocMind_System SHALL log the complete stack trace
4. THE DocMind_System SHALL use appropriate log levels (DEBUG, INFO, WARNING, ERROR)
5. THE DocMind_System SHALL include correlation IDs in logs to trace requests across services
6. THE DocMind_System SHALL write logs to stdout for Docker capture

### Requirement 16: Input Validation

**User Story:** As a security developer, I want all inputs to be validated, so that we prevent injection attacks and corrupt data.

#### Acceptance Criteria

1. THE Go_Service SHALL validate email format in registration and login
2. THE Go_Service SHALL validate minimum password length (8 characters)
3. THE Go_Service SHALL validate MIME types of uploaded files
4. THE Go_Service SHALL validate maximum file size (50 MB)
5. THE Python_Service SHALL sanitize extracted text before generating embeddings
6. WHEN a validation fails, THE DocMind_System SHALL return a 400 error with a descriptive message

### Requirement 17: PDF File Processing

**User Story:** As a user, I want to upload PDF files, so that I can query documents in the most common format.

#### Acceptance Criteria

1. WHEN Python_Service receives a PDF file, THE Python_Service SHALL use a parsing library that supports standard PDF
2. THE Python_Service SHALL extract text from multi-page PDFs
3. THE Python_Service SHALL preserve page order during extraction
4. WHEN a PDF is password-protected, THE Python_Service SHALL return a descriptive error
5. WHEN a PDF contains only images without text, THE Python_Service SHALL return an error indicating no text was found

### Requirement 18: DOCX File Processing

**User Story:** As a user, I want to upload DOCX files, so that I can query Microsoft Word documents.

#### Acceptance Criteria

1. WHEN Python_Service receives a DOCX file, THE Python_Service SHALL use a parsing library that supports OOXML format
2. THE Python_Service SHALL extract text from paragraphs, tables, and lists
3. THE Python_Service SHALL ignore images and graphic elements
4. WHEN a DOCX file is corrupted, THE Python_Service SHALL return a descriptive error

### Requirement 19: Markdown File Processing

**User Story:** As a technical user, I want to upload Markdown files, so that I can query technical documentation.

#### Acceptance Criteria

1. WHEN Python_Service receives an MD file, THE Python_Service SHALL read the content as UTF-8 plain text
2. THE Python_Service SHALL preserve the structure of headers and lists
3. THE Python_Service SHALL include code blocks in the extracted text
4. THE Python_Service SHALL handle different encodings (UTF-8, UTF-16) automatically

### Requirement 20: Resource Cleanup

**User Story:** As a system administrator, I want resources to be released correctly, so that I avoid memory leaks and hanging connections.

#### Acceptance Criteria

1. WHEN Go_Service shuts down, THE Go_Service SHALL close all database connections
2. WHEN Python_Service shuts down, THE Python_Service SHALL close connections to Vector_Store and LLM
3. THE DocMind_System SHALL handle SIGTERM and SIGINT signals for graceful shutdown
4. THE DocMind_System SHALL complete in-progress requests before shutting down (with 30-second timeout)
5. WHEN a service restarts, THE DocMind_System SHALL automatically reconnect to dependencies

### Requirement 21: Configuration Parsing and Pretty Printing

**User Story:** As a developer, I want configuration to be easy to read and modify, so that I can adjust the system without errors.

#### Acceptance Criteria

1. WHEN DocMind_System reads a .env file, THE DocMind_System SHALL parse key-value pairs correctly
2. WHEN a line in .env has invalid format, THE DocMind_System SHALL log a warning and continue
3. THE DocMind_System SHALL ignore empty lines and comments in .env files
4. FOR ALL valid configurations, reading the .env file, modifying a value, writing it, and reading it again SHALL produce equivalent configuration (round-trip property)

### Requirement 22: Service Health Monitoring

**User Story:** As an operator, I want to verify service status, so that I can detect problems quickly.

#### Acceptance Criteria

1. THE Go_Service SHALL expose a /health endpoint that returns 200 when the service is healthy
2. THE Go_Service SHALL verify connectivity to PostgreSQL_DB in the health check
3. THE Go_Service SHALL verify connectivity to Object_Storage in the health check
4. THE Python_Service SHALL expose a gRPC HealthCheck endpoint
5. WHEN a critical dependency is unavailable, THE health check SHALL return 503

### Requirement 23: Result Pagination

**User Story:** As a user with many documents, I want lists to be paginated, so that the interface is responsive.

#### Acceptance Criteria

1. THE Go_Service SHALL support "page" and "limit" query parameters in GET /documents
2. THE Go_Service SHALL return a maximum of 20 documents per page by default
3. THE Go_Service SHALL include pagination metadata in the response (total_count, current_page, total_pages)
4. WHEN a user requests a non-existent page, THE Go_Service SHALL return an empty list with correct metadata

### Requirement 24: Embedding Performance Optimization

**User Story:** As a user, I want document indexing to be fast, so that I can query my documents without waiting long.

#### Acceptance Criteria

1. THE Python_Service SHALL process chunks in batches of 32 for embedding generation
2. THE Python_Service SHALL use asynchronous processing for I/O operations with Vector_Store
3. WHEN Python_Service indexes a large document (>100 chunks), THE Python_Service SHALL report progress every 50 chunks
4. THE Python_Service SHALL cache the embedding model in memory after first load

### Requirement 25: Concurrency Handling

**User Story:** As a multi-user system, I want to handle multiple simultaneous requests, so that all users have a good experience.

#### Acceptance Criteria

1. THE Go_Service SHALL use a PostgreSQL_DB connection pool with minimum 5 and maximum 25 connections
2. THE Python_Service SHALL limit concurrent LLM generation requests to 3 using a semaphore
3. THE Python_Service SHALL queue additional requests when the concurrency limit is reached
4. WHEN multiple users query simultaneously, THE DocMind_System SHALL maintain data isolation by user_id
5. THE DocMind_System SHALL use database transactions for operations that modify multiple records

## Implementation Notes

### Security Considerations
- All passwords must be hashed using bcrypt with cost factor 12
- JWT tokens must be signed using HS256 with a secret of at least 32 characters
- File paths must be validated to prevent path traversal attacks
- Rate limiting should be considered for public endpoints

### Performance Considerations
- Recommended embedding model is "sentence-transformers/all-MiniLM-L6-v2" for its balance between quality and speed
- Qdrant should be configured with HNSW index for fast searches
- MinIO should be configured with replication if high availability is required

### Scalability Considerations
- The system is designed for local execution but can scale horizontally by adding Python_Service replicas
- PostgreSQL can be replaced with a distributed solution if user volume grows significantly
- Vector_Store can be partitioned by user_id ranges to distribute load

### Recommended Tools and Libraries
- Go: gin (web framework), gorm (ORM), jwt-go (JWT), minio-go (S3 client)
- Python: FastAPI, LangChain, sentence-transformers, qdrant-client, PyPDF2, python-docx
- All components are free and open source
